# 云图 一期研发任务拆解

## 1. 文档目的

本文件用于承接 [产品文档/云图-一期上线最小清单.md](产品文档/%E4%BA%91%E5%9B%BE-%E4%B8%80%E6%9C%9F%E4%B8%8A%E7%BA%BF%E6%9C%80%E5%B0%8F%E6%B8%85%E5%8D%95.md)，把首发范围进一步拆成可执行的研发任务，并基于当前产品状态选择合适的一期技术方案。

目标不是设计一套长期最完美的架构，而是选择一套最能支撑首发验证的实现方式。

补充说明：

- 2026-04-28 起，当前一期实现方向已改为直接接入 Supabase，支持真实注册、登录和真实用户数据持久化。
- 因此本文中原先偏向“先不上后端”的判断，后续应以 [产品文档/云图-Supabase接入说明.md](产品文档/%E4%BA%91%E5%9B%BE-Supabase%E6%8E%A5%E5%85%A5%E8%AF%B4%E6%98%8E.md) 和实际代码为准。

## 2. 当前产品状态判断

结合当前 [demo/index.html](demo/index.html) 的实现情况，一期并不是从零开始，而是已经具备以下基础：

- 已有单页交互原型
- 已有星图首页、探索弹层、碎片库、航迹摘要等核心 UI
- 已有 Supabase Auth、登录态恢复与用户状态远端同步
- 已有 localStorage 本地缓存与降级读写逻辑
- 已有基础偏好计分和推荐逻辑
- 已有探索卡与星球数据的前端内嵌结构

这意味着一期不应该直接跳到“前后端分离 + 正式账号体系 + 自建内容后台”的重方案。

当前最合理的方向是：

- 保留静态前端原型的开发速度
- 保留当前已接通的 Supabase Auth 与用户档案快照方案
- 把单文件原型拆成可维护的前端结构
- 补齐内容层、状态层和埋点层
- 让它先成为一个可上线验证留存的产品，而不是先成为一个复杂系统

## 3. 一期推荐技术方案

## 3.1 总体方案

建议采用：

静态站点 + 原生 ES Modules + Supabase Auth 与 `user_profiles.app_state` 状态快照 + 内容数据文件化 + 第三方事件分析

这是基于当前产品形态最现实的一期方案。

原因如下：

1. 当前原型已经是单页静态页面，继续沿用前端直出成本最低。
2. 真实注册、登录和用户数据持久化已经接通，继续沿用现有 Supabase 路线成本最低。
3. 当前交互复杂度不高，不需要为了技术先进性引入 React、Next.js 或完整服务端。
4. 内容以读为主、写为辅，首发阶段完全可以先走文件化内容库，而不是后台 CMS。
5. 埋点分析可以先依赖成熟 SaaS，优先拿到漏斗和留存数据。

## 3.2 前端层方案

建议：

- 继续使用 HTML + CSS + JavaScript
- 从单文件改为多文件结构
- 使用浏览器原生 ES Modules 管理代码

不建议一期直接切 React 或 Vue，原因：

- 当前仓库没有构建链路
- 一期需求主要是整理状态和数据，不是做复杂组件系统
- 重写框架会消耗大量时间在迁移而不是验证产品

推荐目录结构：

```text
demo/
  index.html
  styles/
    main.css
    components.css
  scripts/
    app.js
    state.js
    storage.js
    analytics.js
    render/
      galaxy.js
      modal.js
      shards.js
      route.js
    data/
      systems.js
      cards.js
      themes.js
```

## 3.3 状态与持久化方案

建议：

- 继续使用 Supabase `user_profiles.app_state` 作为一期主持久化方式
- 保留 localStorage 作为本地缓存、离线回退和异常降级手段
- 增加状态版本号与迁移逻辑
- 把 appState 本地读写、远端同步和鉴权状态封装成独立模块

为什么一期仍然采用“用户状态快照”而不是一开始就拆多表：

- 当前要优先验证探索闭环，不是优先做分析型数据建模
- 当前写入模型以单用户进度、碎片和偏好为主，快照结构足够承载
- 在已有 Supabase 基础上继续细拆 `shards`、`events`、`progress` 表，会增加迁移和联调成本

持久化对象建议固定为：

- 用户云端档案状态
- 星球探索状态
- 碎片收藏列表
- 偏好得分
- 最近探索记录
- 引导完成状态
- 埋点事件缓冲区

## 3.4 内容方案

建议：

- 首发探索卡不再写死在主逻辑里
- 把星球数据和探索卡数据拆到独立数据文件
- 用 JS 数据模块维护首发内容库

一期不建议直接接 CMS，原因：

- 当前内容量仍然在 100 到 200 条级别
- 内容生产流程还在定型
- 一期更需要先稳定字段和调度规则，再考虑编辑后台

数据结构最低要求：

- 星球基础信息
- 探索卡完整字段
- 卡片层级与优先级
- 是否已读、是否已收藏所需的 contentId

## 3.5 埋点方案

建议：

- 前端统一封装 trackEvent 方法
- 正式分析先接 PostHog 或 Umami 这类现成工具
- 本地同时保留 debug 事件日志，便于开发期验证

首推 PostHog，原因：

- 静态站点接入快
- 漏斗、留存、事件分析能力够一期使用
- 不需要先自建统计后端

如果暂时不接第三方，也至少要保留统一事件接口，后续接入时不改业务逻辑。

## 3.6 部署方案

建议：

- 一期仍按静态站点部署
- 使用 Cloudflare Pages 或 Vercel 这类静态托管

不建议一期先做 Node 服务端部署，原因：

- 当前没有服务端依赖
- 首发主要目的是快速迭代验证
- 静态托管已经足够承载一期访问量与发布频率

## 4. 一期不建议投入的技术事项

以下事项可以明确不放入一期主开发：

- 自建账号体系
- 自建业务服务端
- 后台内容管理系统
- SSR 或 SEO 重架构
- 前后端分离工程化改造
- TypeScript 全量迁移
- 复杂权限系统
- 复杂分析型拆表与 BI 体系
- 多端同步

这些东西不是没价值，而是不应该抢占一期验证资源。

## 5. 研发任务拆分

以下任务按建议顺序排列，前一项通常是后一项的基础。

### 任务 1. 前端结构拆分

目标：

把当前单文件原型拆成可持续维护的静态前端结构。

具体事项：

- 拆分样式为主样式和模块样式
- 拆分脚本为状态、渲染、交互、数据、埋点模块
- 保持现有视觉和交互不回退
- 确保拆分后仍可直接静态打开和部署

交付物：

- 可运行的多文件前端结构
- 模块入口关系说明

优先级：P0

### 任务 2. 状态层重构

目标：

让探索、收藏、推荐和引导状态具备稳定的读写边界。

具体事项：

- 独立 createDefaultState
- 独立 loadState 和 saveState
- 加入 stateVersion 字段
- 加入旧版本 state 兼容与降级策略
- 统一对外暴露状态更新方法，避免页面逻辑直接散写 appState

交付物：

- 独立状态模块
- 状态字段说明
- 状态迁移规则

优先级：P0

### 任务 3. 内容数据文件化

目标：

让内容库脱离主业务逻辑，便于扩充首发内容。

具体事项：

- 把星球数据抽离为 systems 数据文件
- 把探索卡抽离为 cards 数据文件
- 用 contentId 和 planetId 作为统一索引
- 增加优先级、层级、是否精选等字段
- 明确字段校验规则

交付物：

- 可维护的内容数据文件
- 数据字段约束文档

优先级：P0

### 任务 4. 探索卡调度逻辑补全

目标：

让同一用户的探索结果不显得假随机，也不快速重复。

具体事项：

- 同星球优先未读内容
- 首次探索优先 P1 内容
- 已收藏内容降权
- 最近看过内容短周期内不重复
- 当某星球内容耗尽时给出明确回退策略

交付物：

- 可测试的卡片调度方法
- 调度规则说明

优先级：P0

### 任务 5. 首次探索引导

目标：

让新用户第一次进入时能自然走完首次探索闭环。

具体事项：

- 增加首次进入引导浮层或提示条
- 标注第一步动作
- 标注探索、收藏、继续探索的意义
- 引导状态写入用户档案，避免反复打扰

交付物：

- 新手引导流程
- 引导完成状态持久化

优先级：P0

### 任务 6. 碎片库补齐“回看价值”

目标：

让收藏动作变成个人资产，而不是一次性按钮反馈。

具体事项：

- 增加碎片详情结构统一化
- 补齐来源星球、收藏时间、标签展示
- 增加从碎片返回来源星球继续探索
- 增加按时间倒序和最近收藏展示规则

交付物：

- 可回看、可回流的碎片库

优先级：P0

### 任务 7. 偏好与推荐逻辑整理

目标：

让首页推荐具备最低可解释性。

具体事项：

- 统一探索、收藏对偏好的加权规则
- 区分 topic、mood、planet 三类得分
- 输出推荐说明文案所需的解释字段
- 在首页和航迹页展示推荐理由

交付物：

- 可解释的推荐函数
- 推荐理由展示结构

优先级：P1

### 任务 8. 埋点抽象层接入

目标：

让一期上线后能真实观察转化和回访。

具体事项：

- 封装 trackEvent
- 定义事件名称、字段和触发时机
- 接入第三方分析工具
- 保留开发期本地事件日志
- 补首页访问、点击星球、打开探索、确认探索、收藏碎片、打开碎片库、接受推荐等关键事件

交付物：

- 事件字典
- 埋点实现
- 可验证的埋点触发清单

优先级：P1

### 任务 9. 内容录入与校验工具

目标：

降低首发内容批量扩充时的出错率。

具体事项：

- 定义内容字段模板
- 增加内容文件格式校验脚本
- 检查 contentId 重复、字段缺失、planetId 不匹配
- 为内容团队提供录入示例

交付物：

- 内容录入模板
- 本地校验脚本

优先级：P1

### 任务 10. 发布与回归验证

目标：

确保一期每次发版都不会把核心闭环改坏。

具体事项：

- 制定最小手工回归清单
- 覆盖首次探索、收藏碎片、刷新后保留、推荐更新、碎片回看等关键路径
- 明确本地预览和线上发布步骤

交付物：

- 回归清单
- 发布说明

优先级：P1

## 6. 建议排期

如果按最现实的一期推进节奏，建议拆为 3 个里程碑。

### M1. 可维护原型

包含任务：

- 任务 1
- 任务 2
- 任务 3

目标：

先把当前 demo 从“单文件原型”变成“可继续开发的前端产品骨架”。

### M2. 核心闭环可用

包含任务：

- 任务 4
- 任务 5
- 任务 6
- 任务 7

目标：

让用户真的能完成探索、收藏、回看、继续探索。

### M3. 可上线验证

包含任务：

- 任务 8
- 任务 9
- 任务 10

目标：

让产品不仅能跑，还能被观察、被扩充、被稳定发布。

## 7. 建议角色分工

如果按最小团队配置推进，建议这样分：

- 前端开发：负责结构拆分、状态层、交互、埋点接入
- 内容负责人：负责卡片扩充、来源补齐、字段校验配合
- 产品设计：负责新手引导、推荐解释、碎片回看链路和验收

如果只有 1 名开发，也应优先顺序明确，不要并行分散。

## 8. 一句话结论

云图一期最合理的研发路线，不是重做技术栈，而是在当前静态原型基础上，补出可维护结构、稳定状态、文件化内容和可观察埋点，让它尽快成为一个能真实验证回访意愿的产品。
