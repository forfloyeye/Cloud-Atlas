# 云图 内容系统设计

## 1. 文档定位

本文件用于独立维护云图的内容体系设计，聚焦知识内容如何被定义、组织、抽取、展示、收藏、扩展与运营。

它和概念稿的分工如下：

- 概念稿负责回答“这个产品是什么”
- 本文档负责回答“内容系统如何成立并持续运转”

## 2. 内容系统目标

云图的内容系统，不只是为页面填充素材，而是产品成立的核心基础设施。

它需要同时满足四个目标：

- 让每次探索都有明确的信息价值
- 让不同星球的内容既有统一结构，又有各自气质
- 让收藏、推荐、偏好识别有可计算的数据基础
- 让后续内容运营和扩展有标准可依

## 3. 内容系统在产品中的作用

在云图里，内容不是附属物，而是探索行为真正的“奖励”。

用户之所以愿意继续探索，不只是因为银河界面好看，而是因为每点开一个星系，都可能遇见一个此前不知道、但足够有趣的知识碎片。

因此内容系统承担以下职责：

- 承担探索行为的结果交付
- 支撑探索值增长与收藏沉淀
- 为偏好分析和推荐提供基础标签
- 为未来的星系档案、个人航线、个人星球提供素材基础

## 4. 内容单元定义

云图的最小内容单元建议定义为“探索卡”。

一次探索，对应一张探索卡；一张探索卡，是用户在某个星系中获得的一次知识结果。

### 4.1 探索卡的标准结构

每张探索卡建议统一包含以下字段：

- 标题
- 一句话引子
- 核心知识点
- 延伸碎片
- 标签
- 所属星球
- 文明归属
- 情绪气质
- 难度层级
- 收藏动作

### 4.2 字段说明

#### 标题

作用：

- 作为本次探索内容的第一识别点
- 要足够具体，也要保留一点钩子感

建议：

- 不要写成百科词条名
- 更适合使用“问题式”“反差式”“揭秘式”表达

#### 一句话引子

作用：

- 在用户还没展开正文前，快速建立兴趣
- 决定用户是否愿意继续读完这张卡

建议：

- 控制在 20 到 40 字
- 不解释完，要留下阅读动机

#### 核心知识点

作用：

- 承载这次探索真正的知识价值
- 让用户离开时至少带走一个明确认知

建议：

- 首发阶段以短内容为主，强调信息密度和可读性
- 单卡聚焦一个核心点，不要一张卡塞太多信息

#### 延伸碎片

作用：

- 承担“再多看一点”的轻延展需求
- 为未来推荐、串联探索、碎片库二次浏览提供接口

内容可包括：

- 相关概念
- 相关人物
- 相关器物
- 相关地理空间
- 相关作品或文本

#### 标签

作用：

- 作为偏好识别和推荐的基础数据
- 作为内容筛选与运营分组的基础索引

建议至少包含：

- 题材标签
- 形式标签
- 情绪标签
- 时空标签

#### 所属星球

作用：

- 明确该内容当前属于哪个文明星系
- 用于串联星系内的沉浸感和归属感

#### 文明归属

作用：

- 强化世界观表达
- 让用户感觉是在不同文明中探索，而不只是点击分类

#### 情绪气质

作用：

- 用于统一单个星球的表达口吻
- 也可服务于后续推荐与视觉联动

建议气质维度示例：

- 温柔
- 瑰丽
- 冷峻
- 幽暗
- 奇诡
- 开阔

#### 难度层级

作用：

- 控制内容的阅读门槛
- 支持未来做更精细的内容调度

建议分层：

- L1：轻量冷知识
- L2：完整知识卡
- L3：延展型深读卡

#### 收藏动作

作用：

- 让内容从一次浏览变成用户资产
- 作为用户偏好判断的重要信号

## 5. 内容层级设计

为了兼顾首发效率和后续扩展，建议把内容分为三层。

### 5.1 L1 轻量冷知识卡

特点：

- 信息短
- 钩子强
- 更适合高频探索

适用场景：

- 用户第一次进入某个星球
- 连续探索时作为节奏型内容

### 5.2 L2 完整知识卡

特点：

- 有完整知识点展开
- 兼顾趣味和信息量
- 是首发阶段最主力的内容层级

适用场景：

- 用户对某个星球已有初步兴趣
- 希望提升收藏率和停留时长

### 5.3 L3 延展型深读卡

特点：

- 信息密度更高
- 更适合在用户已形成偏好后投放
- 可作为未来星系档案页的重要内容来源

适用场景：

- 已探索较深的用户
- 作为某个题材的解锁奖励或延展阅读

## 6. 内容生产原则

### 6.1 先有趣，再准确，再完整

云图不是教材，也不是论文数据库。

首发阶段内容的优先级应该是：

1. 先让用户想看
2. 再让用户看懂
3. 最后再考虑是否足够完整

### 6.2 一张卡只讲一个点

不要试图在一张卡里讲完一个庞大主题。用户第一次探索获得的应该是一个抓得住的点，而不是一个压过来的知识包。

### 6.3 每个星球都要有自己的“说话方式”

同样是知识卡，不同星球的表达气质必须不同。

例如：

- 手工星人星球更温柔、贴手、生活化
- 克苏鲁潮汐星球更低语、诡异、不可直视
- 大秦观测站更冷峻、硬质、制度感强
- 敦煌飞天星球更流动、明艳、带空间感

### 6.4 避免把内容写成普通科普站

要避免：

- 过度平铺直叙
- 没有叙事感的罗列
- 全是定义，没有发现感

用户想要的不是“被告知”，而是“被带去看见”。

## 7. 首发阶段内容组织方式

### 7.1 按星球组织

首发阶段建议以星球为一级组织方式，每个星球下放具体探索卡。

### 7.2 按标签横向连接

在星球之外，再通过标签建立横向关联，例如：

- 神话想象
- 帝国制度
- 民间手艺
- 怪奇生命
- 世界构建

这样同一个用户即使在不同星球之间，也能形成兴趣轨迹。

### 7.3 按难度做节奏控制

对新用户优先投放 L1 和 L2 内容，对高兴趣用户逐步释放 L3 内容。

## 8. 收藏与偏好信号

内容系统不仅要承载阅读，还要承载信号采集。

建议重点记录以下行为：

- 点击了哪个星球
- 打开了哪张探索卡
- 是否读完
- 是否确认探索结果
- 是否收藏为星际碎片
- 是否继续探索同一星球
- 是否切换到相邻题材星球

这些行为会比用户口头选择“我喜欢什么”更真实。

## 9. 首发内容准备建议

每个首发星球建议先准备：

- 30 到 50 张探索卡
- 其中 L1 占比约 40%
- L2 占比约 50%
- L3 占比约 10%

这样 10 个星球首发期至少需要 300 张以上可用内容，才能让云图有“真的可以反复探索”的体感。

## 10. 后续可扩展方向

内容系统后续可以继续扩展为以下子模块：

- 内容字段规范
- 标签体系设计
- 星球口吻规范
- 推荐规则草案
- 内容审核与纠错机制
- 内容编辑工作流
- AI 辅助生成与人工校对流程

## 11. 一句话总结

云图的内容系统，本质上是在把知识做成可探索、可收藏、可推荐、可成长的文明碎片网络。

## 12. 字段规范

本章节用于把前文的“探索卡结构”进一步落成可执行标准，方便后续建表、建库、录入内容和做筛选推荐。

### 12.1 字段设计原则

- 字段要先服务首发版本，不预埋过多未来字段
- 同一字段只承载一种判断，不混用
- 先保证内容编辑可执行，再考虑算法精细度
- 能通过枚举约束的字段，尽量不要放任自由填写

### 12.2 探索卡字段总表

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `content_id` | string | 是 | 内容唯一 ID，用于内容库主键 |
| `title` | string | 是 | 探索卡标题 |
| `hook` | string | 是 | 一句话引子 |
| `body` | text | 是 | 核心知识点正文 |
| `extra_fragments` | array | 否 | 延伸碎片列表 |
| `planet_id` | string | 是 | 所属星球 ID |
| `planet_name` | string | 是 | 所属星球名称 |
| `civilization` | string | 是 | 文明归属或题材文明 |
| `difficulty_level` | enum | 是 | 内容层级，L1/L2/L3 |
| `topic_tags` | array | 是 | 题材标签 |
| `format_tags` | array | 是 | 形式标签 |
| `mood_tags` | array | 是 | 情绪气质标签 |
| `era_tags` | array | 否 | 时代标签 |
| `region_tags` | array | 否 | 地域标签 |
| `character_tags` | array | 否 | 人物标签 |
| `artifact_tags` | array | 否 | 器物或对象标签 |
| `reading_time` | integer | 否 | 预估阅读时长，单位秒 |
| `source_note` | string | 否 | 来源备注或资料依据 |
| `status` | enum | 是 | 草稿、待审、可发布、下线 |
| `is_featured` | boolean | 否 | 是否作为精选内容 |
| `created_at` | datetime | 是 | 创建时间 |
| `updated_at` | datetime | 是 | 更新时间 |

### 12.3 字段详细约束

#### `content_id`

建议规则：

- 全局唯一
- 不依赖标题生成，避免后续标题修改影响数据稳定性
- 建议使用“星球缩写 + 数字序号”形式

示例：

- `craft-001`
- `myth-023`
- `qin-014`

#### `title`

建议规则：

- 控制在 12 到 28 字之间
- 优先使用问题式、反差式、揭秘式表达
- 避免写成纯词条或纯论文题目

可用方向：

- 为什么……
- 原来……
- 某个看起来……的东西，其实……

#### `hook`

建议规则：

- 控制在 20 到 40 字
- 只负责勾兴趣，不抢正文工作
- 不要把答案一次说完

#### `body`

建议规则：

- L1 建议 80 到 180 字
- L2 建议 180 到 400 字
- L3 建议 400 到 900 字
- 一张卡只讲一个知识点，必要时拆卡

内容结构建议：

- 先给结论或反差
- 再解释原因或背景
- 最后给一个能让人记住的收束点

#### `extra_fragments`

建议规则：

- 1 到 3 条即可
- 每条延伸碎片保持短句形式
- 只做引导，不做正文重复

#### `planet_id` 和 `planet_name`

建议规则：

- `planet_id` 保持稳定，用于系统关联
- `planet_name` 用于展示，可迭代命名
- 首发阶段一个内容只归属于一个主星球，避免多归属导致管理混乱

#### `civilization`

建议规则：

- 表示该内容所在的文明母体，不等同于标签
- 一个内容只保留一个主文明归属

示例：

- 手工文明
- 非遗文明
- 上古神话文明
- 丝路文明
- 帝国文明

#### `difficulty_level`

枚举值：

- `L1`
- `L2`
- `L3`

建议规则：

- 首发阶段不要再细分更高层级
- 难度不是学术难度，而是阅读负担和信息密度的综合判断

#### 各类标签字段

建议规则：

- 每张卡至少有 2 个 `topic_tags`
- 每张卡至少有 1 个 `format_tags`
- 每张卡至少有 1 个 `mood_tags`
- 其余标签按内容需要补充，不强制凑满

#### `reading_time`

建议规则：

- 可以由字数自动估算
- 首发阶段用于前端展示和节奏判断，不参与复杂排序

#### `source_note`

建议规则：

- 不直接面向用户强展示
- 主要用于内部编辑追溯和校对
- 如果是高争议内容，必须记录来源依据

#### `status`

枚举值：

- `draft`
- `reviewing`
- `published`
- `archived`

使用建议：

- 录入完成后先进入 `draft`
- 经人工校对后改为 `reviewing`
- 可正式上线的内容改为 `published`
- 不再投放但保留历史记录的内容进入 `archived`

### 12.4 首发阶段最小必填集

为了提高录入效率，首发阶段每张探索卡至少必须填完以下字段：

- `content_id`
- `title`
- `hook`
- `body`
- `planet_id`
- `planet_name`
- `civilization`
- `difficulty_level`
- `topic_tags`
- `format_tags`
- `mood_tags`
- `status`

如果这些字段都完整，内容卡已经具备上线与基础推荐能力。

## 13. 标签体系设计

标签系统不是给内容“贴关键词”，而是把内容变成可连接、可筛选、可推荐的网络。

### 13.1 标签设计原则

- 标签少而稳，优先保证可维护性
- 先建立主干标签，再逐步扩展长尾标签
- 同一标签只表达一个维度
- 标签命名尽量避免近义词重复和范围重叠

### 13.2 首发标签采用六维结构

首发阶段建议把标签分为以下六类：

1. 题材标签
2. 形式标签
3. 情绪标签
4. 时代标签
5. 地域标签
6. 对象标签

### 13.3 题材标签

作用：

- 这是推荐和偏好判断中最重要的一组标签
- 用于表达“这条内容本质上在讲什么”

首发建议标签池：

- 手工技艺
- 民间工艺
- 非遗传承
- 节俗仪式
- 上古神话
- 异兽传说
- 佛教艺术
- 丝路交流
- 宇宙恐惧
- 怪奇生命
- 帝国制度
- 历史人物
- 古代工程
- 奇妙发明
- 世界构建
- 文化母题

### 13.4 形式标签

作用：

- 用于表达内容的呈现类型或知识切入口

首发建议标签池：

- 冷知识
- 设定拆解
- 人物故事
- 器物观察
- 地理想象
- 图像细节
- 制度解析
- 误解纠偏
- 对照比较
- 起源溯源

### 13.5 情绪标签

作用：

- 用于表达阅读时的氛围感和星球气质
- 后续可和视觉推荐联动

首发建议标签池：

- 温柔
- 治愈
- 瑰丽
- 神秘
- 奇诡
- 幽暗
- 冷峻
- 宏大
- 明亮
- 荒诞
- 潮湿
- 静谧

### 13.6 时代标签

作用：

- 用于表达内容发生或对应的主要时间背景

首发建议标签池：

- 上古
- 先秦
- 秦代
- 汉唐
- 古代
- 近现代
- 架空时代
- 跨时代

### 13.7 地域标签

作用：

- 用于表达内容的主要地理来源或文化空间

首发建议标签池：

- 中原
- 西域
- 敦煌
- 山海世界
- 深海
- 森林
- 多元宇宙
- 丝绸之路

### 13.8 对象标签

作用：

- 用于表达内容主要涉及的人、物、生命或文本对象

首发建议标签池：

- 异兽
- 壁画
- 洞窟
- 器物
- 乐舞
- 帝王
- 官僚系统
- 菌类
- 机械装置
- 禁书
- 英雄
- 手工材料

### 13.9 标签命名规范

命名时建议遵守以下规则：

- 尽量使用 2 到 6 字短语
- 不混用口语和学术语体
- 不用“其他”“杂项”作为常规标签
- 同一维度不允许并列近义词重复存在

示例：

- 可以用“帝国制度”，不要同时再有“王朝制度”“帝制系统”三个近义标签
- 可以用“奇妙发明”，不要再单独出现“奇怪发明”作为同层标签

### 13.10 标签使用规则

每张探索卡建议遵守以下配比：

- 题材标签 2 到 3 个
- 形式标签 1 到 2 个
- 情绪标签 1 到 2 个
- 时代标签 0 到 2 个
- 地域标签 0 到 2 个
- 对象标签 0 到 3 个

避免的问题：

- 标签过多，导致内容失去重心
- 标签过少，导致推荐和筛选无效
- 所有卡都贴同一组大词，失去区分度

## 14. 内容卡录入模板

为了方便后续批量建内容池，建议统一使用以下模板。

```yaml
content_id: dunhuang-001
title: 飞天为什么看起来像在空中漂浮，而不是在飞
hook: 你以为飞天的重点是“飞”，其实古人更在意的是身体如何摆出一种失重的美感。
body: |
	敦煌飞天最迷人的地方，不只是“会飞”，而是那种仿佛不受地面约束的姿态。画师通过飘带、身体扭转和衣褶走向，制造出一种比真实飞行更轻盈的视觉错觉。它并不是现代意义上的空气动力学，而是一种宗教想象和审美经验共同塑造出来的“升腾感”。
```
